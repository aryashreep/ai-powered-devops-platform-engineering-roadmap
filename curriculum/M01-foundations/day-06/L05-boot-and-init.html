Linux Boot Sequence

Jab tak ye chain na samjho, boot failure diagnose nahi hogi!

BIOS (Legacy)

  • MBR — first 512 bytes of disk
  • Max 2 TB disk support
  • 16-bit real mode
  • No Secure Boot ✗
VS

UEFI (Modern)

  • ESP partition (FAT32)
  • GPT — 9.4 ZB support
  • 64-bit protected mode
  • Secure Boot support ✓

systemd — PID 1 Ka Raaj

systemd is always PID 1. Har doosra process iska child. Ye units manage karta hai — .service, .target, .timer, .socket.

.service
Long-running process
.target
Group (= runlevel)
.timer
Cron replacement
# /etc/systemd/system/myapp.service
[Unit]
Description=My App
After=network.target

[Service]
ExecStart=/usr/bin/myapp
Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target

journalctl — Log Superpower

Har service ka output binary journal mein. Syslog se pehle ke logs bhi. Reboots survive karta hai. Ye seekhna compulsory hai.

journalctl -bCurrent boot ke logs
journalctl -b -1PREVIOUS boot — crash analysis ke liye ←
journalctl -b -u nginxIs boot mein ek specific service
journalctl -p errErrors aur above only
journalctl -f -u nginxLive tail — tail -f ka bada bhai

# Start / Stop / Restart
systemctl start   nginx
systemctl stop    nginx
systemctl restart nginx

# BEST PRACTICE: enable + start ek saath!

systemctl disable --now nginx

# Unit file edit ke baad ZAROOR yeh karo
systemctl daemon-reload

Step 3 — journalctl Debug Karo

# Current boot ke logs
journalctl -b

# PICHLE boot ke logs (crash ke baad analysis)
journalctl -b -1

# Ek service ke logs
journalctl -b -u nginx

# Sirf errors
journalctl -p err -b

# Live follow
journalctl -f -u nginx

Step 4 — Apna Service Banao

# 1. Script banao
cat > /usr/local/bin/hello.sh <<'EOF'
#!/bin/bash
while true; do echo "$(date): hello"; sleep 5; done
EOF
chmod +x /usr/local/bin/hello.sh

# 2. Unit file banao
cat > /etc/systemd/system/hello.service <<'EOF'
[Unit]
Description=Hello World Service
After=network.target
[Service]
ExecStart=/usr/local/bin/hello.sh
Restart=on-failure
[Install]
WantedBy=multi-user.target
EOF

# 3. Deploy karo
systemctl daemon-reload
systemctl enable --now hello.service

# 4. Verify karo
systemctl status hello.service
journalctl -f -u hello.service
⑤ Common Mistakes · 5 min
⚠ Ye Galtiyan Mat Karna!
Golu
"Main yahi karta tha Jagu — daemon-reload bhool jaata tha!"
① daemon-reload bhoolna
vi /etc/systemd/system/myapp.service systemctl restart myapp # OLD unit se restart!
vi /etc/systemd/system/myapp.service systemctl daemon-reload # ← zaroori! systemctl restart myapp
② enable karna par start bhoolna
systemctl enable nginx # Sirf NEXT REBOOT pe start hoga # Abhi service DEAD hai — koi nahi jaanta!
systemctl enable --now nginx # Enable + Start ek saath ✓
③ /var/log/syslog mein dhundhna
tail -f /var/log/syslog # Pre-syslog messages miss, prev boot miss
journalctl -b -u myapp # Full journal, is boot journalctl -b -1 -u myapp # Previous (crashed) boot
④ restart vs reload confusion
systemctl restart nginx # Sab connections kill → downtime!
systemctl reload nginx # SIGHUP → config re-read → zero downtime ✓
⑤ Restart= directive nahi likhna
[Service] ExecStart=/usr/bin/myapp # crash → forever dead
[Service] ExecStart=/usr/bin/myapp Restart=on-failure RestartSec=5s # Self-healing ✓
⑥ Interview Questions · 5 min
❓ Interview Mein Yeh Poochenge
1
PID 1 kya hai aur kyun matter karta hai?
▾
PID 1 = systemd — kernel ke baad sab se pehle start hone wala process. Sab doosre processes iske children hain. Agar PID 1 crash ho — kernel panic. Containers mein bhi PID 1 = entrypoint — exit kare toh container band.
2
systemctl enable aur systemctl start mein kya fark hai?
▾
start — service abhi start karo, reboot pe survive nahi karega.

enable — symlink create karo next boot ke liye, par abhi start nahi hoga.

Best: systemctl enable --now — dono ek saath. Common bug: enable bina start → next reboot pe fail, koi nahi jaanta.
3
Service reboot ke baad start nahi hui. Kaise debug karoge?
▾
1. systemctl status <svc> — quick overview + exit code
2. journalctl -b -u <svc> — is boot ke poore logs
3. journalctl -b -1 -u <svc> — pichle boot ke logs
4. systemd-analyze verify /etc/systemd/system/<svc>.service — syntax check
5. ExecStart path exist karta hai? Executable permissions hain?
4
initramfs kya hai aur kyun chahiye?
▾
Kernel ko disk read karne ke liye drivers chahiye, par drivers disk pe hain — chicken-and-egg. initramfs ek tiny compressed root filesystem hai jo bootloader memory mein load karta hai. Isme drivers hote hain jo real root mount karne ke liye kaafi hain. Disk encryption, LVM, RAID — sab yahan handle hote hain.
5
BIOS vs UEFI — DevOps engineer ko kyun matter karta hai?
▾
UEFI + Secure Boot = supply chain security. Unsigned bootloaders load nahi ho sakte — Kubernetes node hardening (CKS) ke liye relevant. Cloud VM images build karte waqt GRUB config UEFI vs BIOS pe alag hogi — yeh jaanna boot failures bachata hai.

Boot Sequence — Quick Ref
StepKya hota hai
1. Power ONHardware POST, CPU init
2. BIOS/UEFIBoot device dhundho, bootloader load karo
3. GRUB2Boot menu, kernel + initramfs RAM mein load
4. KernelHardware detect, initramfs ko temp root mount
5. initramfsReal root mount karne ke liye drivers (LVM, crypto)
6. Real Rootinitramfs se actual / partition pe pivot
7. systemd PID 1default.target padho, services parallel mein start
8. Your ServicesApp start, login prompt
