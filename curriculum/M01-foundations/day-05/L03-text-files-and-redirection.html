What we learn today

Reading large files, redirecting output, building text pipelines, and in-place editing with sed — together these form a Linux engineer's daily toolkit.

🎯 Learning Objectives
  • Read large files without crashing your terminal — cat, head, tail, less
  • Redirect output to files, append, split streams — >, >>, 2>&1, /dev/null

💡 Why it matters: Because every DevOps task — log analysis, config management, CI pipelines — runs on text files. Without these commands, your infrastructure is blind.

📄 Reading Files — Choose the Right Tool

cat
Small files only (configs, scripts). Prints the entire file at once. Never use on a 2GB file.
head -20
First N lines. Useful for checking CSV headers or verifying file format. Default is 10 lines.
tail -f
Live follow. Watch log files in real-time. The #1 incident response tool. Exit with Ctrl+C.
less
Paginate large files. Search with /pattern, n for next match, q to quit. Memory-safe.
# cat — small configs only
cat /etc/hostname
cat /etc/hosts

# head — first N lines
head -20 /var/log/syslog

# tail — last N lines, or LIVE follow
tail -50 /var/log/syslog
tail -f /var/log/syslog          # live stream

# less — safe for any size
less /var/log/syslog              # /ERROR to search, q to quit

↪ Redirection — Control Your Streams

I/O Stream Flow

stdin (0) → Command → stdout (1) → file / pipe
stderr (2) → 2>&1 → stdout (1) or /dev/null 
# > overwrite (DANGEROUS — wipes the file!)
echo "hello" > output.txt

# >> append (SAFE — adds to the end of the file)
echo "world" >> output.txt

# | pipe — one command's output is another's input
cat /etc/passwd | grep "root"

# tee — write to stdout AND file simultaneously
tail -f app.log | tee live_capture.txt

# 2>&1 — merge stderr into stdout
ls /nonexistent 2>&1 | tee errors.log

# /dev/null — discard output (silence)
cron_script.sh > /dev/null 2>&1

🔎 Searching & Text Processing

# grep — pattern search
grep -i "error" app.log         # case insensitive
grep -r "DB_HOST" /etc/          # recursive directory search
grep -v "DEBUG" app.log          # invert — lines WITHOUT pattern
grep -c "ERROR" app.log          # count matching lines

# wc — count lines, words, bytes
wc -l app.log                   # count lines only
grep "ERROR" app.log | wc -l   # count ERROR lines

# sort — sort lines
sort /etc/hosts                  # alphabetical
sort -k2 data.csv               # sort by field 2
sort -rn counts.txt             # reverse numeric

# uniq — remove/count duplicates (requires sorted input)
sort ips.txt | uniq            # deduplicate
sort ips.txt | uniq -c        # count occurrences

# cut — extract fields
cut -d: -f1 /etc/passwd        # field 1 using : delimiter (usernames)
cut -d, -f2,4 data.csv         # fields 2 and 4 from CSV

✎ sed — Stream Editor

sed treats text files as a stream — read, transform, output. Perfect for mass updates to config files.

# s/old/new/g — global substitution
sed 's/localhost/db.prod.internal/g' config.env

# -i — in-place edit (modifies the file directly)
sed -i 's/DEBUG/INFO/g' app.conf

# -i.bak — in-place edit WITH backup (best practice)
sed -i.bak 's/old_url/new_url/g' nginx.conf

# /pattern/d — delete matching lines
sed '/^#/d' config.conf          # delete comment lines
sed '/^$/d' config.conf          # delete blank lines

# Multiple operations with -e
sed -e 's/foo/bar/g' -e '/^#/d' input.txt

⑥ Common Mistakes · 5 min
⚠ Don't Make These Mistakes!
Golu
"I was using > thinking it appends, Jagu — the whole file got wiped!"
① > vs >> confusion — overwrite vs append
echo "new entry" > important.log # WIPES entire file!
echo "new entry" >> important.log # Appends safely ✓
② cat huge_file — terminal freeze
cat /var/log/syslog # 500MB file — terminal freezes!
less /var/log/syslog # Paginated, memory-safe ✓ tail -100 /var/log/syslog # Last 100 lines only ✓
③ sed -i without backup
sed -i 's/old/new/g' nginx.conf # No backup — how do you undo?
sed -i.bak 's/old/new/g' nginx.conf # .bak backup auto-create ✓ # Revert: mv nginx.conf.bak nginx.conf
⑦ Interview Questions · 5 min
❓ What You'll Be Asked in Interviews
1
What does the Linux pipe (|) do? Explain with an example.
▾
The pipe connects stdout of one command to stdin of another — without creating temporary files. Example: cat file | grep "ERROR" | sort | uniq -c — 4 commands run in a pipeline, data flowing through memory. Unix philosophy: small tools, each doing one thing well, composed with pipes.
2
grep -v vs plain grep — what's the difference?
▾
grep pattern — only print lines that MATCH the pattern.

grep -v pattern — only print lines that do NOT match the pattern. Invert match. Use case: filter out noise — grep -v "DEBUG" app.log — remove DEBUG lines, see everything else.
3
What's the difference between > and >>?
▾
> — output redirection, overwrite. If the file exists, it WIPES it; otherwise creates it. Dangerous for existing files.

>> — output redirection, append. If the file exists, it ADDS to the end; otherwise creates it. ALWAYS use >> for log files and important data.
