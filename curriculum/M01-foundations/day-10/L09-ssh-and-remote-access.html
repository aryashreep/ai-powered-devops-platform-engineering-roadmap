③ Theory Group 1 · 5 min
🔑 Password vs Key-Based Auth
50K
Brute-force attempts in 2 days
2^128
ED25519 key space (impossible)
1 line
PasswordAuthentication no

Password Auth

  • Human-memorizable — also human-guessable
  • Brute-forceable — tools like hydra, medusa
  • Transmitted over network (even if encrypted)
  • Rotation pain — must update everywhere
  • Shared passwords = no individual audit
VS

Key-Based Auth

  • Mathematically unforgeable (ED25519 = 128-bit)
  • Private key NEVER transmitted — only signature
  • Revoke one key = one user, not everyone
  • Brute force: physically impossible
  • Add passphrase = 2nd security layer

💡 How Challenge-Response Works

Client Server | --- "I'm user golu" -------------------> | | <-- "Prove it: here's a random challenge" -- | | --- Signs challenge with private key -------> | | <-- "Signature matches public key! Welcome" -- | Private key stays local. Only the signature goes over the wire.

The private key never leaves your machine. Server only checks the math — did this signature come from the matching private key? No password, no brute-force vector.

④ Theory Group 2 · 5 min
🔐 ssh-keygen — RSA vs ED25519

Generate Your Key Pair

# RECOMMENDED: Generate ED25519 key (fast, secure, small key size)
ssh-keygen -t ed25519 -C "golu@company.com"
# Saves to: ~/.ssh/id_ed25519       (private — NEVER share this!)
#           ~/.ssh/id_ed25519.pub   (public  — this goes on server)

# Always set a passphrase when prompted!
# Enter passphrase (empty for no passphrase): *** use a strong one ***

# Legacy RSA (4096 bit — for compatibility with old systems only)
ssh-keygen -t rsa -b 4096 -C "golu@company.com"
# Saves to: ~/.ssh/id_rsa,  ~/.ssh/id_rsa.pub

# View your public key (safe to share)
cat ~/.ssh/id_ed25519.pub
# Output: ssh-ed25519 AAAA...long-base64-string... golu@company.com

Algorithm Comparison

Algorithm Key Size Speed Security Use Case
ED25519 256 bits Very fast 128-bit equivalent Modern (recommended)
RSA 4096 4096 bits Slower 128-bit equivalent Legacy compatibility
RSA 2048 2048 bits Moderate 112-bit (minimum) Minimum acceptable
RSA 1024 1024 bits — BROKEN Do NOT use
⑤ Theory Group 3 · 5 min
📄 authorized_keys, known_hosts & ~/.ssh/config

authorized_keys (Server Side)

This file lives on the server at ~/.ssh/authorized_keys. It lists all public keys allowed to log in as this user.

# Manual: append public key to server's authorized_keys
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys  # CRITICAL: must be 600 or SSH rejects it!
chmod 700 ~/.ssh/                  # Directory must be 700

# Automated: ssh-copy-id (recommended — handles permissions automatically)
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server

# Verify it was added
cat ~/.ssh/authorized_keys

known_hosts (Client Side)

This file lives on your client machine at ~/.ssh/known_hosts. Remembers every server's fingerprint — MITM protection!

# Auto-added on first connect with a prompt:
# "The authenticity of host '...' can't be established."
# "Are you sure you want to continue connecting? (yes/no)"

# If server key CHANGES → WARNING (possible MITM attack!)
# WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!

# Remove stale/old key (e.g. after server rebuild)
ssh-keygen -R hostname
ssh-keygen -R 192.168.1.10

~/.ssh/config — Multi-Server Productivity

Instead of typing ssh -i ~/.ssh/key ubuntu@192.168.1.10 -p 22 every time, define host aliases:

# ~/.ssh/config

Host prod-web
    HostName     192.168.1.10
    User         ubuntu
    IdentityFile ~/.ssh/id_ed25519
    Port         22

Host bastion
    HostName     bastion.company.com
    User         ec2-user
    IdentityFile ~/.ssh/company_key

Host db-server
    HostName     10.0.0.5
    User         admin
    ProxyJump    bastion    # Jump through bastion host automatically!

# Usage after config: just type the alias
ssh prod-web           # = ssh -i ~/.ssh/id_ed25519 ubuntu@192.168.1.10
ssh db-server          # = ssh through bastion → then to 10.0.0.5
⑥ Theory Group 4 · 5 min
🔌 SSH Tunneling & Port Forwarding

💡 Analogy: Secure Tube Through the Internet

SSH tunnel = ek encrypted tube jo kisi bhi port ka traffic andar se carry kar sakti hai.
Local forwarding: remote database ko apne laptop pe lao.
Remote forwarding: apni local dev app ko public server pe expose karo.

Port Forwarding Commands

# LOCAL port forwarding: access remote service as if it's local
# Access remote MySQL (port 3306) on your local machine
ssh -L 3306:localhost:3306 user@db-server
# Now connect: mysql -h 127.0.0.1 -P 3306 (traffic flows through SSH)

# Another example: local:8080 → remote:80
ssh -L 8080:localhost:80 user@webserver
# Browse: http://localhost:8080 → sees the remote server's port 80

# REMOTE port forwarding: expose local service on remote server
ssh -R 8080:localhost:3000 user@public-server
# Now: public-server:8080 → your local machine's port 3000

# DYNAMIC forwarding (SOCKS5 proxy — route ALL traffic)
ssh -D 1080 user@gateway-server
# Configure browser to use SOCKS5 proxy at localhost:1080

# Keep connection alive (prevent timeout)
ssh -o ServerAliveInterval=60 user@server

# Background tunnel (-N = no command, -f = background)
ssh -N -f -L 3306:localhost:3306 user@db-server

Local Port Forwarding Flow

Your Machine → localhost:3306 → SSH Encrypted Tunnel → db-server → MySQL :3306

Database never exposed to internet. All traffic encrypted inside SSH.

⑧ Common Mistakes · 3 min
🚫 Top Mistakes & How to Fix Them
✕ Mistake 1 — Using deprecated/weak key types
ssh-keygen -t rsa -b 1024 # RSA 1024: CRACKED / insecure ssh-keygen -t dsa # DSA: deprecated, removed from OpenSSH 9.0 ssh-keygen -t rsa -b 2048 # Minimum acceptable, not ideal
ssh-keygen -t ed25519 -C "email@example.com" # Always ED25519 in 2026
✕ Mistake 2 — Wrong permissions on ~/.ssh files
chmod 777 ~/.ssh/ # SSH will REFUSE to use keys chmod 644 ~/.ssh/authorized_keys # Too permissive — SSH rejects this chmod 644 ~/.ssh/id_ed25519 # Private key exposed to others!
chmod 700 ~/.ssh/ # Directory: owner only chmod 600 ~/.ssh/authorized_keys # Authorized keys: owner read/write chmod 600 ~/.ssh/id_ed25519 # Private key: owner read/write only chmod 644 ~/.ssh/id_ed25519.pub # Public key: readable by all (OK)
✕ Mistake 3 — Disabling password auth BEFORE testing key auth
# DO NOT do this without verifying key login works first! echo "PasswordAuthentication no" >> /etc/ssh/sshd_config systemctl restart sshd # Now if your key fails → you are LOCKED OUT forever
# Correct sequence: # 1. Copy key to server # 2. Test key login works: ssh -i ~/.ssh/id_ed25519 user@server # 3. Open a SECOND session and keep it alive # 4. THEN disable password auth + restart sshd # 5. Test from a NEW terminal — only then close the old session
⑨ Interview Questions · 5 min
🌳 Interview Q & A
1
How does SSH key authentication work? Explain the challenge-response flow.
▼
The client sends its public key identity to the server. The server checks if that public key exists in ~/.ssh/authorized_keys. If found, the server generates a random challenge, encrypts it with the public key, and sends it to the client. Only the holder of the matching private key can decrypt it. The client signs the decrypted challenge with the private key and sends back the signature. The server verifies the signature using the public key — if it matches, access is granted. The private key never leaves the client machine.
2
What is the difference between authorized_keys and known_hosts?
▼
authorized_keys lives on the server — it lists public keys of all clients allowed to log in as that user. known_hosts lives on the client — it stores the fingerprints of servers the client has connected to before. known_hosts protects against Man-in-the-Middle attacks: if a server's fingerprint changes unexpectedly, SSH warns you that the server may have been compromised.
3
How do you set up access from a jump/bastion host to a private server?
▼
In ~/.ssh/config, use the ProxyJump directive. Example: Host private-db; HostName 10.0.0.5; ProxyJump bastion. Then ssh private-db will automatically SSH to the bastion first, then from bastion SSH to the private server — all in one command, transparently. The old way was ProxyCommand with netcat, but ProxyJump is simpler and recommended.
4
Why is ED25519 preferred over RSA 2048?
▼
ED25519 uses elliptic curve cryptography (Curve25519), which achieves 128-bit security with only a 256-bit key — far smaller than RSA 4096 which needs 4096 bits for similar security. ED25519 is significantly faster for both key generation and signature operations. The keys are tiny (68 chars vs 3200+ for RSA 4096), making them easy to store and transmit. It's also immune to several side-channel attacks that can affect RSA implementations. RSA is only preferred when connecting to very old systems that don't support ED25519.
⑩ Quick Quiz · 2 min
🤹 Test Karo Apne Aap Ko
What permissions must ~/.ssh/authorized_keys have?

Answer Explained:

600 (owner read+write, no one else) is correct. SSH is extremely strict — if authorized_keys has any group or world permissions (644, 666, 777), SSH will refuse to use it. 400 (read-only) also works but is unnecessarily restrictive — you need write permission to add more keys. Same rule: ~/.ssh/ directory must be 700.
