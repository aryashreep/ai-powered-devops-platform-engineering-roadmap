③ Theory · 12 min
👤 Group 1: useradd / usermod / userdel

💡 Analogy: HR Department

useradd: HR ne naya employee register kiya — ID card banai, desk assign ki (home dir).
usermod: HR ne employee ka department change kiya, ya designation update ki.
userdel: HR ne employee ko exit kiya — records hata diye. 

# Create user with home directory and bash shell
sudo useradd -m -s /bin/bash username
sudo passwd username

# Create user with specific UID and comment
sudo useradd -m -s /bin/bash -u 1501 -c "Tokyo Dev" tokyo

# Modify user — add to supplementary group (APPEND!)
sudo usermod -aG developers username

# Lock account (disable login without deleting)
sudo usermod -L username   # lock — puts ! in /etc/shadow
sudo usermod -U username   # unlock

# Change user's shell
sudo usermod -s /sbin/nologin www-data

# Delete user (keep home directory)
sudo userdel username
# Delete user AND home directory + mail spool
sudo userdel -r username

⚠ Critical Warning: -G vs -aG

WRONG — Replaces ALL existing groups!
sudo usermod -G developers tokyo # tokyo loses docker, sudo, and all other groups!
RIGHT — Appends to existing groups
sudo usermod -aG developers tokyo # tokyo keeps all existing groups, developers added
👥 Group 2: groupadd / groupmod / gpasswd
# Create a group
sudo groupadd developers
sudo groupadd -g 1500 dbadmins   # with specific GID

# Add user to group (via gpasswd)
sudo gpasswd -a username developers   # add member
sudo gpasswd -d username developers   # remove member

# Set group administrator
sudo gpasswd -A tokyo developers   # tokyo can manage group members

# Rename group
sudo groupmod -n devteam developers

# Check group members
getent group developers   # developers:x:1001:tokyo,berlin,golu
groups username           # show all groups for user
id username               # uid, gid, groups — full info

User → Groups → Resources

tokyo (user) → developers (group) → /opt/dev-project (770)
berlin (user) → developers + admins → sudo access

Groups = scalable access control. 100 users, 1 group change = done!

🔒 Group 3: passwd / su / sudo / sudoers
# Change your own password
passwd

# Root changes another user's password
sudo passwd username

# Force password change on next login
sudo passwd -e username

# Switch to another user (keeps current environment)
su username

# Switch to root with full login environment
su -          # full login shell — loads root's .bashrc, PATH etc

# Run single command as root
sudo command

# Run command as another user
sudo -u www-data command

# Open root shell (equivalent to sudo su -)
sudo -i       # or: sudo su -

# Edit sudoers SAFELY — validates syntax before saving!
sudo visudo

sudoers File Format — /etc/sudoers

# Syntax: user  HOST=(run_as_user:run_as_group)  commands

# Full sudo for a user
ubuntu  ALL=(ALL:ALL)  ALL

# Group with no-password docker command
%developers  ALL=(ALL)  NOPASSWD: /usr/bin/docker

# tokyo can only restart nginx, no password
tokyo  ALL=(ALL)  NOPASSWD: /usr/bin/systemctl restart nginx

# DANGEROUS — avoid in production!
# username ALL=(ALL) NOPASSWD: ALL

# Drop-in files (preferred over editing sudoers directly)
# /etc/sudoers.d/developers

Always use sudo visudo — never edit /etc/sudoers directly. A syntax error can lock you out of sudo!

su — Switch User

  • Requires the target user's password
  • su username — partial env switch
  • su - — full login environment
  • No audit trail in sudoers log
  • Avoid sharing root password ⚠
VS

sudo — Superuser Do

  • Uses YOUR own password
  • Granular rules per user/group/command
  • Full audit log in /var/log/auth.log
  • Timeout after 5 min (configurable)
  • Production standard ✓
📄 Group 4: /etc/passwd / /etc/shadow / /etc/group

/etc/passwd — User Database (world-readable)

Format: username : x : UID : GID : comment : home : shell

# /etc/passwd — NOT passwords (moved to /etc/shadow)
# username:x:UID:GID:comment:homedir:shell

root:x:0:0:root:/root:/bin/bash
www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin
tokyo:x:1001:1001:Tokyo Dev:/home/tokyo:/bin/bash
berlin:x:1002:1002:Berlin Dev:/home/berlin:/bin/bash

# x = password stored in /etc/shadow
# /usr/sbin/nologin = service account, no interactive login

/etc/shadow — Password Hashes (root-only, 640)

Format: username : hash : lastchange : min : max : warn : inactive : expire

# /etc/shadow — readable only by root
# username:$algo$salt$hash:lastchange:min:max:warn:inactive:expire:

tokyo:$6$rounds=5000$salt$hashhash...:19500:0:90:7:::
#      ^^^ $6 = SHA-512, $5 = SHA-256, $1 = MD5 (obsolete)
#                                        ^^^^ days since Jan 1 1970
#                                             ^^^ 0 = no min age
#                                               ^^^^ 90 = max days
#                                                  ^^^ warn 7 days before

# Set password expiry policy
sudo chage -M 90 username    # max 90 days
sudo chage -l username        # list policy

# Locked account has ! prefix
lockeduser:!$6$hash...:19500:0:99999:7:::

/etc/group — Group Database (world-readable)

Format: groupname : x : GID : member1,member2,...

# /etc/group
# groupname:x:GID:members

root:x:0:
sudo:x:27:berlin,professor
developers:x:1001:tokyo,berlin
admins:x:1002:berlin,professor
docker:x:998:tokyo,berlin

# Quick lookups
getent group developers     # developers:x:1001:tokyo,berlin
id tokyo                    # uid=1001(tokyo) gid=1001(tokyo) groups=1001(tokyo),1002(developers),998(docker)
groups tokyo                # tokyo : tokyo developers docker
④ Best Practices · 3 min
🛡 Security Best Practices

Production-Grade Security Rules

  • Never give direct root SSH — use sudo + individual accounts. Every action is logged with the real username.
  • Use groups for resource access — not individual ACLs. 100 devs, 1 group rule = maintainable.
  • Set password expiry: chage -M 90 username — enforce rotation policy.
  • Lock service accounts: usermod -s /sbin/nologin www-data — prevent interactive login.
  • Use sudoers.d drop-in files — never edit /etc/sudoers directly, always visudo.
  • Audit sudo access regularly: grep -r 'ALL' /etc/sudoers /etc/sudoers.d/

⑥ Common Mistakes · 5 min
⚠ Teen Common Mistakes
Mistake 1 — sudo su - vs sudo -i (environment differences)
sudo su - # spawns new shell, PAM modules may not run
sudo -i # proper interactive root — runs PAM, loads full env
Mistake 2 — usermod -G vs usermod -aG (CRITICAL!)
sudo usermod -G developers tokyo # REPLACES all groups! tokyo loses docker, sudo etc
sudo usermod -aG developers tokyo # APPENDS. -a = append. Always use -aG!
Mistake 3 — Shared root password vs individual sudo
ssh root@server # Who did what? No audit trail. Password shared.
ssh tokyo@server # then: sudo command → logs show tokyo ran it # /var/log/auth.log: tokyo : TTY=pts/0 ; COMMAND=/bin/rm ...
⑦ Interview Questions
🎯 Interview Prep
1
What is the difference between usermod -G and usermod -aG?
▼
usermod -G groupname user REPLACES all supplementary group memberships — the user loses every group except the one specified. usermod -aG groupname user APPENDS the new group while keeping all existing ones. In production, always use -aG. Forgetting the -a flag has caused incidents where users suddenly lost access to docker, sudo, and other critical groups.
2
How do you grant a user sudo access for only one specific command?
▼
Edit sudoers with sudo visudo (or create a file in /etc/sudoers.d/) and add: tokyo ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx. The full path to the command is required. You can add multiple commands comma-separated. Using visudo is mandatory — it validates syntax before saving, preventing lockouts.
3
What is /etc/shadow and why is it separate from /etc/passwd?
▼
/etc/passwd is world-readable (permissions 644) because many programs need to look up usernames by UID. If password hashes were in /etc/passwd, anyone could run offline dictionary attacks. /etc/shadow (permissions 640, owned by root:shadow) stores the actual hashed passwords, last change date, expiry policy, and account status. Only root and members of the shadow group can read it. This separation is called "shadow password suite" and has been standard since the 1980s.
4
How do you check which groups a user belongs to?
▼
Three commands: (1) groups username — simple list; (2) id username — full details including UID, primary GID, and all supplementary groups; (3) getent group | grep username — shows all groups where the user appears as a member. Note: if a user was added to a group while logged in, they need to log out and back in (or run newgrp groupname) for the new group to take effect in the current session.
